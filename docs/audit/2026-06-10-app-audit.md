# MacroPlan — Full Product & Technical Audit

**Date:** 2026-06-10
**Scope:** Full codebase review, build/test verification, live render of guest pages, UI screenshot review (qa-*.png), API integration deep-dive.
**Baseline health:** `tsc --noEmit` clean · 142/142 unit tests pass · production build succeeds (with env vars present).

---

## 1. What works well from a user's perspective

- **The batch-prep core loop is genuinely differentiated.** "Tell us your prep day, macros, and containers → get a batch cooking plan with a cooking-order timeline" is a real, specific promise that Eat This Much / Prospre don't make. The generate form (`/meal-plans/generate`) is four fields and one button — excellent.
- **Onboarding is clean.** Six steps, proper validation (Zod), lossless lb↔kg conversion, keyboard accessible, and a satisfying macro reveal at step 6 backed by Mifflin-St Jeor + TDEE math that is correct and unit-tested.
- **Honest marketing.** No fabricated testimonials; claims ("3s generation", "100% macro-calculated") are defensible; 9 real blog posts with substantive content; FAQ answers real objections.
- **Real engineering quality.** RLS on every table, server-side free-tier enforcement (`app/actions/batch-prep.ts:36`), quotas for favorites/swaps, skeleton loaders, error/loading/not-found routes on recipes, `prefers-reduced-motion` respected, no console.log litter, no lorem ipsum.
- **Meal logging works.** Dashboard macro rings are fed by a persisted `logged_meals` table, not client state.

## 2. Does it feel premium and worth paying for?

**Verdict: the core is worth $9.99/mo; the first impressions are not yet.** Score ≈ 7/10.

What breaks the premium illusion, in the order a new user hits it:

1. **Signup wall before any value.** Middleware (`middleware.ts:29-37`) forces account creation before onboarding. The landing CTA promises "Start your first prep — free" and immediately demands an email. The guest-onboarding infrastructure (Zustand + localStorage + migration) exists but is dead behind this wall. This is the single biggest conversion killer.
2. **"No image" recipe grid.** ~70% of recipe cards render a grey placeholder (see §4 for root cause). A food app where the food has no photos reads as broken.
3. **Dashboard app chrome leaks onto marketing pages.** Logged-in users see the dashboard sidebar on `/pricing`, `/help`, `/terms` (`components/layout/app-layout-wrapper.tsx:97` uses a hardcoded pathname blocklist missing those routes).
4. **Brand identity is split three ways**: "Macro Plan" in app chrome (10+ files), "MacroPlan" in metadata/landing, and the repo itself is misspelled "MacoPlan".
5. **Placeholder blog/OG images** — all 9 posts use `placehold.co` URLs (`lib/blog-data.ts`), so every social share looks unfinished.
6. **Small trust dents**: plan card shows "Daily Meal Plan - 08/01/2026" for a Jan 8 plan (date locale bug), generic gradient-icon tiles instead of food imagery on plan cards, emoji-wave greeting, zero social proof on landing.

## 3. Core offering vs. bloat

**The core (keep, polish, market):**
1. Macro calculation engine (`lib/calculations/`) — correct, tested.
2. Claude-powered batch-prep generation (`lib/services/batch-prep-*`) with training/rest-day split.
3. Prep-day cooking timeline + shopping list + PDF export.
4. Meal logging against daily targets (dashboard).
5. Stripe subscription with enforced free-tier limits.

**Bloat / debt (cut or consolidate):**
1. **Three overlapping list systems**: legacy `shopping_lists` (built from pre-pivot `meal_plans`/`meal_plan_meals`), a separate `grocery-lists` feature (`app/grocery-lists/[id]`), and the batch-prep plan's embedded `shopping_list` JSONB. One product needs one list.
2. **The entire legacy meal-plan system.** The April 2026 pivot doc (`docs/plans/2026-04-09-batch-prep-p0-design.md`) declared the Recipe-API generator replaced, yet `app/actions/meal-plans.ts`, `app/actions/plans.ts`, and parts of `/meal-plans/[id]` still serve old-style plans. Two parallel plan systems confuse the UI ("Daily Meal Plan" cards next to batch preps) and the codebase.
3. **TOTP 2FA** (`otplib`, `qrcode`, `lib/security/totp.ts`, verify page) — enterprise-grade auth ceremony for a pre-launch meal app. It also crashes the build when env vars are missing. Email + Google OAuth is enough until there are users to protect.
4. **Dead dependencies**: `isomorphic-dompurify` (0 imports), `convert-units` (1 file), `framer-motion` (2 files — replaceable with CSS), stale FatSecret types in `lib/types/database.ts` / `lib/types/recipe.ts`.
5. **Stale docs**: CLAUDE.md documents deleted services (`meal-plan-generator.ts`) and a guest onboarding flow the middleware forbids. Misleading for every future contributor (human or AI).

The recipe browser is the borderline case: it supports the "500+ recipes" claim and feeds favorites, but it is the least differentiated surface and currently the most visibly broken (images). It's a supporting act — it should not receive more investment than the batch-prep loop.

## 4. Is the API set up correctly, and is the app displaying meals well?

**Recipe-API.com — correctly set up.** Lazy key init (no build-time crash), `X-API-Key` header, SHA-256-hashed cache keys, 7-day search / 30-day detail TTLs in Supabase, in-flight request deduplication, errors propagated and mapped to generic client messages. Minor: cached search results lose original ranking (`.in()` doesn't preserve order) and ignore stored pagination edge cases.

**Anthropic (batch prep) — well-engineered.** Current model (`claude-sonnet-4-6`), structured-JSON parse with Zod, macro-accuracy check (±tolerance), one retry with a correction hint, then best-of-two rather than hard failure — the right call for LLM macro math. **Bug:** success paths log `{input_tokens: 0, output_tokens: 0}` to `anthropic_usage_log` (`lib/services/batch-prep-generator.ts:99,116,124`) — real usage from `response.usage` is discarded, so cost observability is blind. No per-request timeout or per-user generation cap beyond the free-tier gate.

**Unsplash — the broken link, and the cause of the "No image" grid.**
- Free tier = 50 requests/hour. A search page fetches images for up to 20 uncached recipes *synchronously inside the search action* (`app/actions/recipe-search.ts:57`), so 2–3 fresh page views exhaust the hour.
- Failures return `null` and are **not negative-cached** (`lib/services/unsplash.ts:132-134`), so every subsequent page view re-attempts and re-fails with 403 — placeholders persist indefinitely.
- Even when it works, a generic `"{recipe name} food"` stock photo is usually *not the actual dish* — borderline misleading for a food product.

**Meal display:** recipe detail pages are strong (hero, macro card, full nutrition facts, log + favorite buttons, loading/not-found states). Batch-prep plan views (timeline, containers, shopping list) are the best screens in the app. The recipes *grid* is the weak point purely because of images; macro chips and card layout are fine. The grocery list page hardcodes `bg-gray-50` (`app/grocery-lists/[id]/page.tsx`), breaking dark mode tokens.

**Deploy fragility:** `next build` fails in any environment without Supabase env vars (prerender of `/login/verify-2fa` constructs a client at build time). Middleware uses the service-role key and performs a `user_profiles` query on **every** authenticated request — a latency tax and an unusually broad use of the most powerful key.

**Schema reproducibility:** only 5 migrations exist (2026-03-09 onward). The founding tables (`user_profiles`, `recipes`, `meal_plans`, `logged_meals`, favorites, caches) have no migrations in the repo — the schema lives only in production Supabase and cannot be rebuilt from source.

## 5. Is it slop? What must change to get users on the platform

**It is not slop.** It's a coherent product with a smart positioning pivot, real tests, and honest copy — wearing a broken first-impression layer. Ranked by impact on acquiring users:

1. **Unblock the funnel (highest ROI).** Let guests run onboarding steps 1–6 and *see their macros* before the signup wall; gate plan generation/saving instead. The entire infrastructure (Zustand persistence + `/onboarding/complete` migration) already exists — this is a middleware change, not a feature build.
2. **Fix food imagery.** Short term: negative-cache Unsplash misses, fetch images out-of-band (after responding), default the browse tab to `mustHaveImages`. Real fix: own the images — a one-time batch job generating a consistent image per cached recipe beats live stock-photo lookups in cost, speed, and honesty.
3. **Show the product on the landing page.** The hero mockup is good; add a real prep-day timeline screenshot and 2–3 genuine user quotes the moment they exist. Replace placeholder blog/OG images.
4. **Separate marketing from app.** Route groups (`(marketing)` / `(app)`) with server layouts instead of the client-side pathname blocklist — kills the sidebar leak class of bugs permanently and removes auth-state flicker.
5. **One plan system, one list system.** Finish the pivot: migrate/retire legacy `meal_plans` surfaces and collapse the three list features into the batch-prep shopping list.
6. **Sweat the trust details.** One brand spelling everywhere, fix the date formatting bug, food imagery on plan cards, commit baseline schema migrations, update CLAUDE.md to the post-pivot architecture.
7. **Tighten time-to-aha.** Signup → first generated batch plan should be one unbroken session; the dashboard's "Generate" nudge is good — make it the default landing state for plan-less users.
